# 一、Hyperliquid 深度分析

## 1. 产品定位与核心价值主张

### 1.1 解决的核心问题

Hyperliquid 的诞生源于一个根本性矛盾: **中心化交易所(CEX)拥有优秀的交易体验但存在托管风险和不透明性,而传统去中心化交易所(DEX)虽然自托管安全但性能极差、流动性不足。** Hyperliquid 试图通过构建专用 L1 区块链,将 CEX 级别的性能(亚秒级延迟、每秒20万订单处理)与链上透明性和自托管安全性结合起来。

核心价值主张:
- **完全链上的订单簿**: 每一笔订单、撤单、成交、清算都透明地发生在链上,具有一区块最终性(one-block finality)
- **零 Gas 费交易**: 所有交易操作(下单、撤单、止盈止损、TWAP)均为签名操作,包含在共识中,用户无需支付 Gas
- **CEX 级性能**: 中位延迟 0.1 秒,P99 延迟 0.5 秒,吞吐量 200,000 订单/秒
- **自托管**: 用户资产始终由自己的钱包控制,无需信任第三方

### 1.2 目标用户

| 用户类型 | 具体画像 |
|---------|---------|
| **专业量化交易者** | 需要低延迟 API、复杂订单类型(TWAP、缩放订单)、子账户体系 |
| **高频永续合约交易者** | 从 CEX 迁移,追求 50x 杠杆 + 链上透明 |
| **DeFi 原生用户** | 使用 HyperEVM 上的借贷、质押、收益策略 |
| **做市商/流动性提供者** | 通过 HLP Vault 或自建 Vault 获取做市收益 |
| **RWA/传统资产交易者** | 交易代币化股票、大宗商品、外汇永续合约(2026年新增) |

### 1.3 竞争定位

Hyperliquid 在永续合约 DEX 市场占据**绝对统治地位**:
- **市场份额**: 占全部 DEX 永续合约未平仓量的 ~70%
- **月交易量**: ~$2080 亿(2026年3月),而 dYdX 仅 $250-300 亿
- **TVL**: ~$42-62 亿(数据源差异),远超 dYdX 的 $3-4 亿
- **日活跃交易者**: 231,000(2026年3月创历史新高)

与主要竞品对比:

|------|------------|------|-----|--------------|
| 架构 | 自建 L1 + 订单簿 | Cosmos L1 + 订单簿 | Arbitrum + AMM | Solana + AMM |
| 日交易量 | $50-120 亿 | $5-10 亿 | $2-5 亿 | $3-8 亿 |

## 2. 技术架构深度分析

### 2.1 HyperBFT 共识机制

HyperBFT 是 Hyperliquid 自研的共识算法,灵感来源于 HotStuff 及其后续迭代,但针对端到端延迟进行了深度优化。

**核心工作原理:**
- 基于拜占庭容错(BFT),可容忍最多 1/3 的恶意验证者
- 采用领导者轮换机制: 领导者提议区块,验证者通过多轮通信达成共识
- **关键优化**: 与 Tendermint/以太坊不同,HyperBFT 允许在当前区块哈希计算完成前就开始排序下一批交易。即验证者 1 可以同时宣布"我批准交易 A 且已准备好处理交易 B",大幅减少等待时间
- 投票权重与质押量(包括委托质押)成正比,采用 DPoS 模型

**性能指标:**

| 指标 | 数值 |
|------|------|
| 中位延迟 | 0.1 秒 |
| P99 延迟 | 0.5 秒 |
| 订单吞吐量 | 200,000 订单/秒 |
| 最终性 | 1 区块(~0.2秒) |

### 2.2 双层架构: HyperCore + HyperEVM

Hyperliquid 的状态执行被分为两个组件:

**HyperCore (交易层)**
- 包含完全链上的永续合约和现货订单簿
- 所有订单、撤单、成交、清算都透明地在链上发生
- 继承 HyperBFT 的一区块最终性
- 交易完全免 Gas -- 操作以签名动作形式包含在共识中
- 区块内订单优先级: (1) 撤单 > (2) Post-Only 订单 > (3) GTC 订单 > (4) IOC 订单。这意味着做市商可以在同一区块内先撤回旧报价,再被 Taker 订单执行之前更新

**HyperEVM (智能合约层)**
- 提供完全兼容以太坊的通用智能合约平台
- 与 HyperCore 原生互通,可无权限地访问 HyperCore 的流动性和金融基础设施
- 支持标准 Solidity 开发,降低 DeFi 开发者迁移成本
- 2025年3月启用 HyperCore 与 HyperEVM 之间的代币转移功能

### 2.3 订单撮合引擎

- 完全链上的中央限价订单簿(CLOB)
- 价格-时间优先撮合
- 支持原生的批量处理和优先级排序
- 做市商友好设计: Post-Only 订单在 Taker 之前处理

### 2.4 验证者集

| 指标 | 详情 |
|------|------|
| 当前验证者数量 | 16 个(从最初 4 个增加) |
| 活跃验证者集上限 | 24 个(按质押量排名) |
| 基金会质押占比 | ~81% |
| 共识达成时间 | ~2 分钟(JELLY 事件暴露) |

### 2.5 结算机制

- 所有交易以 USDC 结算
- 一区块最终性,无需等待多区块确认
- 原生桥接通过 Arbitrum 上的智能合约管理存取款
- Bridge2 升级后支持更多链的直接存款

## 3. 交易产品功能全景

### 3.1 永续合约

**支持交易对:**
- **加密货币**: 150+ 对,涵盖 BTC、ETH 及各类山寨币
- **传统资产(RWA via HIP-3)**: 原油、黄金、白银、S&P 500(通过 S&P Dow Jones Indices 授权)、预上市股权永续(SpaceX、Anthropic)
- **外汇**: 主流货币对
- 截至2026年3月,平台前30大市场中仅7个是纯加密对,RWA 交易量已占总量约 30%

**杠杆设置:**

| 资产类型 | 最大杠杆 |
|---------|---------|
| BTC / ETH | 50x |
| 主流山寨币 | 20-25x |
| 小市值代币 | 3-5x |
| 大宗商品/股票永续 | 视品种而定 |

杠杆上限与订单大小和类型(市价 vs 限价)相关,由清算所确定性执行。

### 3.2 现货交易

- 基于 HIP-1 原生代币标准的完全链上订单簿
- 无需许可的代币上架: 通过荷兰拍卖获取 Ticker -> 部署 HIP-1 代币 -> 设置创世余额 -> 创建 USDC 交易对
- HIP-2 Hyperliquidity: 内置于 HyperCore 的自动化双边流动性方案,使用链上订单簿(非传统 AMM 流动性池)
- 现货 Taker 费率 0.070%,Maker 费率 0.040%

### 3.3 保证金模式

| 模式 | 说明 |
|------|------|
| **全仓保证金(Cross Margin)** | 使用全部永续合约余额作为所有仓位的担保品,一个仓位的亏损可由其他仓位覆盖 |
| **逐仓保证金(Isolated Margin)** | 每个仓位的担保品独立限定,风险隔离 |

### 3.4 订单类型

Hyperliquid 提供媲美 CEX 的丰富订单类型:

| 订单类型 | 说明 |
|---------|------|
| **市价单(Market)** | 以当前最优价格立即执行 |
| **限价单(Limit - GTC)** | 挂在订单簿上直到成交或取消 |
| **Post-Only (ALO)** | 仅挂单不吃单,保证 Maker 费率 |
| **IOC (Immediate or Cancel)** | 未立即成交部分自动取消 |
| **止盈(Take Profit)** | 价格触及目标时触发 |
| **止损(Stop Loss)** | 价格触及止损位时触发 |
| **止损限价(Stop Limit)** | 停止价触发后以限价单形式执行 |
| **缩放订单(Scaled Orders)** | 在价格范围内自动分布多个限价单 |
| **TWAP** | 大额订单拆分为小单,每30秒执行一次,单笔最大滑点3%,若子订单未完全成交则后续子单变大(上限为正常子单3倍) |
| **Reduce Only** | 仅用于减仓,防止反向开仓 |

### 3.5 子账户系统

- 支持创建多个子账户
- 所有子账户交易量计入主账户总量
- 所有账户共享同一费率等级
- 适合策略隔离和风险管理

### 3.6 Vault 系统

**HLP (Hyperliquidity Provider) Vault:**

> ⚠️ **重要区分**: HLP **不是** GMX/Jupiter 那样的 AMM 流动性池。Hyperliquid 是纯订单簿模型，HLP 本质上是一个**协议管理的自动化做市基金** — 它在订单簿上像普通做市商一样挂限价单赚价差，同时负责接管清算仓位。用户存入 USDC 相当于出资给这个"去中心化做市团队"，分享做市收益和承担做市风险。

- 功能: 在订单簿上双边挂单做市、执行清算接管、累积交易手续费
- 储备金: ~$4 亿
- 任何用户都可以存入 USDC 参与 HLP,分享做市和清算收益
- 由多个子策略组成,包括清算者 Vault(Liquidator Vault)

**Assistance Fund (AF):**
- 独立于 HLP,储备约 $13 亿
- 接收 97% 的协议费用
- 每日自动用 USDC 在公开市场回购 HYPE 代币

**用户 Vault:**
- 任何用户均可创建自定义交易 Vault
- 其他用户可存入资金跟随策略
- 类似"链上基金"

### 3.7 资金费率机制

- 每小时计算和结算
- 基于永续合约价格与现货指数价格的偏差
- 多头支付空头(或反之),使永续合约价格锚定现货
- 是 Delta 中性策略(如 Liminal 协议)的收益来源

### 3.8 清算机制

**清算触发条件:**
- 当交易者的维持保证金率低于阈值(通常 0.5%)时触发清算

**清算流程(三层防御):**

1. **常规清算**: 市场上的普通清算流程
2. **HLP Backstop 清算**: 维持保证金低于 2/3 的仓位由 HLP 的清算者 Vault 接管
3. **自动减仓(ADL)**: 最后手段,仅在常规清算和 HLP 均失败时触发。当交易者账户价值为负(产生坏账)时,对手方按未实现盈利和杠杆排名,以前一个预言机价格强制平仓

## 5. 用户体验分析

### 5.1 Web UI 设计哲学

- **极简专业风格**: 深色主题,信息密度高,面向专业交易者
- 类似 CEX 的交易界面布局: K线图 + 订单簿 + 交易面板 + 仓位管理
- 集成 TradingView 图表

- 实时 PnL 显示和仓位管理

### 5.2 移动端体验

- 通过 PWA (渐进式 Web 应用) 提供移动端访问
- 响应式设计适配手机/平板
- 暂无原生 iOS/Android 应用

### 5.3 API/SDK 开发者支持

| 资源 | 详情 |
|------|------|
| **REST API** | 完整的信息查询和交易执行端点 |
| **WebSocket** | 实时市场数据和账户更新 |
| **Python SDK** | 官方维护,GitHub: hyperliquid-dex/hyperliquid-python-sdk(需 Python 3.10) |

- `Info`: 公共只读数据,无需 API 密钥
- `Exchange`: 需要认证和签名的交易操作

### 5.4 入金流程

**主要入金路径:**
1. **Arbitrum 原生桥(官方默认)**: 仅支持 Arbitrum 上的 USDC 存入
2. **跨链桥(2026年扩展)**:
   - Across Protocol: 支持 20+ 条链(Ethereum、Solana、Base、Polygon 等)的 USDC/USDT
   - Jumper Exchange
   - LayerZero 桥
   - Symbiosis
3. **Solana 直接存入**: 2026年新增,支持直接从 Solana 钱包存入 SOL
4. 所有主流桥默认将资产发送到 HyperCore

### 5.5 支持的链

- **存款**: Arbitrum(官方)、Ethereum、Solana、Base、Polygon、Optimism 等 20+ 条链(通过第三方桥)
- **提款**: 主要通过 Arbitrum 桥提取 USDC

### 6.2 HIP 提案体系

| 提案 | 内容 |
|------|------|
| **HIP-1** | 原生代币标准,部署即获得链上现货订单簿 |

### 6.3 Builder Codes / 推荐系统

**推荐计划:**
- 需完成 $10,000 交易量后可创建推荐码
- 被推荐者前 $2500万交易量享 4% 费率折扣
- 推荐人获得被推荐者费用的 10%(扣除折扣)
- 推荐奖励适用于前 $10亿交易量

**Builder Codes:**
- 允许构建者对其代发的订单收取费用
- 每单独设,灵活度极高
- 用户需通过主钱包签名批准每个 Builder 的最大费率
- Builder 需持有至少 100 USDC 的永续账户价值
- 适用于永续合约双边,现货仅适用于卖方

### 6.4 HYPE 代币经济学

**回购 + 销毁机制:**
- Assistance Fund 接收 97% 协议费用,每日自动在公开市场回购 HYPE
- 三大收入来源: 永续合约费用(主力)、HIP-1 拍卖费(100%回购)、现货费(USDC 部分回购,HYPE 部分直接销毁)
- 实际效果(2026年3月27日示例): 回购销毁 34,496 HYPE(均价 ~$38.51),发放质押奖励 26,784 HYPE,**净减少 7,711 HYPE** -- 已实现净通缩

**质押:**
- 2024年12月30日上线
- DPoS 模型,年化收益率与总质押量平方根成反比(类以太坊设计)
- 质押可获得 5%-40% 的交易费率折扣

## 7. 风险与争议

### 7.1 JELLY 事件 (2025年3月)

**事件经过:**
1. 一名交易者在 Hyperliquid 上建立了 JELLY 代币的大额空头仓位
2. 同时在链上(其他交易所)大量买入 JELLY,推动价格在1小时内暴涨 **429%**
3. 空头仓位被清算,HLP Vault 被迫接管

## 2. 技术架构

### 2.1 JLP(Jupiter Liquidity Pool)模型

JLP 是 Jupiter Perps 的核心架构,**本质上是一个由多资产组成的流动性池,充当所有交易者的统一对手方**。这与 Hyperliquid/dYdX 的订单簿模型有根本区别。

**工作原理:**
1. LP 将资产存入 JLP 池,获得 JLP 代币作为份额凭证
2. 交易者开仓时,从 JLP 池中借入资产(多头借入标的资产,空头借入稳定币)
3. 交易以预言机价格执行,不存在滑点(大额交易有价格影响费)
4. 交易者盈利 = JLP 池亏损;交易者亏损 = JLP 池收益
5. JLP 额外赚取开仓/平仓费、借款费等交易费用

**JLP 池组成(目标权重):**

| 资产 | 目标权重 | 当前权重(约) |
|------|---------|-------------|
| SOL | 44% | 44.54% |
| ETH | 10% | 9.67% |
| WBTC | 11% | 9.84% |
| USDC | 26% | 26.78% |
| USDT | 9% | 9.17% |

**JLP 收益来源:**
- 开仓/平仓费(75% 归 JLP)
- 借款费(75% 归 JLP,其中 10% 为协议储备)
- 价格影响费
- 交易者净亏损(统计上交易者群体整体亏损)
- 2025年8月起: 池内闲置 SOL 的质押收益

### 2.2 预言机系统

Jupiter Perps 采用**多预言机冗余架构**:

| 预言机 | 角色 |
|--------|------|
| **Dove Oracle** | 主预言机,由 Chaos Labs(Edge Pricing Data)与 Jupiter 联合设计,计算效率极高,可在单笔交易中同时更新全部 5 个资产的价格 |
| **Pyth Network** | 参考价格校验(Sanity Check)+ 备用预言机,当 Dove 价格过时时自动切换 |
| **Chainlink** | 额外的价格数据源,增强定价弹性 |

这种多预言机设计的优势在于: 即使某个预言机被操纵,其他预言机仍然可以提供正确价格,降低了类似 Hyperliquid JELLY 事件的风险。

### 2.3 Keeper 网络

Jupiter Perps 的交易执行采用**两步 Keeper 模型**:

1. **第一步**: 交易者提交请求交易(Request Transaction)到 Solana 链上
2. **第二步**: Keeper 监控链上请求,抓取最新预言机价格,执行请求并作为独立交易提交

Keeper 的职责包括:
- 监控并执行开仓/平仓请求
- 监控止盈/止损(TP/SL)触发条件,当标记价格达到设定水平时关闭仓位
- 执行清算操作
- 更新预言机价格

### 2.4 订单匹配机制

**Jupiter Perps 不使用传统订单簿。** 这是与 Hyperliquid/dYdX 的根本区别:

- 交易者的对手方是 **JLP 池**,不是其他交易者
- 所有交易以**预言机价格**执行,无买卖价差
- 不存在 Maker/Taker 概念,所有人都是对 JLP 池交易
- 大额交易通过**价格影响费**(Price Impact Fee)模拟订单簿滑点
- 流动性深度取决于 JLP 池的规模和各资产的利用率

### 2.5 Solana 上的延迟与吞吐量

| 指标 | 数值 |
|------|------|
| 交易确认时间 | ~0.4 秒 |
| Solana 出块时间 | ~400ms |
| 理论 TPS | 65,000 TPS(Solana 全网) |
| Keeper 执行延迟 | 通常 1-2 个 Solana slot(0.8-1.2 秒) |

由于 Keeper 两步模型,实际交易延迟比纯链上订单簿方案(如 Hyperliquid)要长,但仍远优于以太坊 L2 方案。

### 2.6 智能合约架构

- 基于 Solana 的 **Anchor 框架**编写
- 核心合约为 Jupiter Perpetuals Program,部署在 Solana 主网
- 合约管理: JLP 池状态、仓位管理、清算逻辑、费用计算
- 已通过多方审计
- 闭源(核心交易逻辑未完全开源)

## 3. 交易产品功能

### 3.1 支持的永续合约交易对与最大杠杆

Jupiter Perps 目前支持约 **50 个永续合约交易对**,核心三大对:

| 交易对 | 最大杠杆 | 备注 |
|--------|---------|------|
| SOL-PERP | 250x | 旗舰对,流动性最深 |
| BTC-PERP | 250x | 2025年升级至 250x |
| ETH-PERP | 250x | 2025年升级至 250x |
| 其他对(如 JUP, WIF 等) | 5x - 100x | 较新增,流动性较浅 |

需要注意: 虽然理论最大杠杆为 250x,但实际可用杠杆受到 JLP 池中对应资产利用率的限制。当某资产利用率过高时,借款费会急剧上升,实际交易成本显著增加。

### 3.2 订单类型

| 订单类型 | 说明 |
|---------|------|
| **市价单(Market Order)** | 以当前预言机价格立即执行 |
| **限价单(Limit Order)** | 设定价格,由 Keeper 在价格到达时执行 |
| **止盈(Take Profit)** | 设定盈利目标价格自动平仓 |
| **止损(Stop Loss)** | 设定亏损限价自动平仓 |
| **OCO(One-Cancels-Other)** | TP 和 SL 配对,一个触发另一个自动取消,共享保证金 |
| **OTO-OCO** | 父触发单先执行,成功后自动激活子 TP/SL 对 |

V2 触发订单升级(2025年): 从池比例计算改为 **USD 价格触发**,用户设定美元价格,Jupiter 自动处理转换和路由执行。V2 订单默认链下存储且隐私,触发前不可见。

### 3.3 保证金模式

Jupiter Perps 当前采用**逐仓保证金(Isolated Margin)**模式:
- 每个仓位的保证金独立隔离
- 一个仓位被清算不影响其他仓位
- 用户可以对单个仓位增减保证金
- 暂不支持全仓保证金(Cross Margin)

### 3.4 仓位管理功能

- **调整杠杆**: 开仓后可动态调整杠杆倍数
- **增减保证金**: 向仓位追加或提取保证金
- **部分平仓**: 支持部分关闭仓位
- **TP/SL 绑定**: 开仓时可同时设置止盈止损

### 3.5 JLP Vault 详细分析

**池规模与里程碑:**
- 2025年9月: TVL 突破 **$20 亿**
- 当前 APY: ~17.58%(2025年9月数据)
- 历史费用分配: 仅2025年1月至今已向 JLP 持有者分配超 **$1.37 亿**
- 年化费用收入: 约 $5.1 亿(基于周 $980 万费用推算)

**收益分配机制:**
- 总费用的 75% -> JLP 池
- 总费用的 12.5% -> JUP 持有者(2025年2月17日起)
- 总费用的 12.5% -> 协议金库 + JUP 持有者

**JLP 价格历史:**
- 上线价格(2024年2月): ~$2.20
- 历史最高价(2025年10月): ~$5.99
- 当前价格: ~$3.68
- 上线以来总回报: ~+67%(价格层面,含费用再投资)

**JLP 风险:**
- **交易者对手方风险**: 当交易者群体大幅盈利时(如单边行情),JLP 会遭受损失
- **无常损失(IL)**: 标的资产价格波动导致池内资产比例偏离目标权重
  - 示例: 2025年12月某周,JLP 相对其篮子资产产生 4.25% 的无常损失
  - 但多数周份 IL 可控(如2026年2月仅 0.04%)
- **资产集中度风险**: SOL 占比 44%,高度依赖 SOL 价格表现

### 3.6 资金费率/借款费机制

**Jupiter Perps 不使用传统资金费率(Funding Rate),而是采用借款费(Borrow Fee)模型。** 这是其与 CEX 及 Hyperliquid/dYdX 的关键区别。

| 特征 | Jupiter Perps(借款费) | 传统模型(资金费率) |
|------|----------------------|-------------------|
| 方向 | **始终为正**(交易者始终付费) | 可正可负(多头付空头或反之) |
| 支付对象 | 交易者 -> JLP 池 | 多头 <-> 空头 |
| 计算方式 | 基于借入金额和池利用率 | 基于多空持仓比例差异 |
| 复利 | 每小时复利 | 通常每 8 小时结算 |
| 费率(基础) | SOL/ETH/BTC: 0.008%/小时; USDC/USDT: 0.01%/小时 | 动态浮动 |

**借款费公式:**
```
每小时借款费 = (总锁定代币 / 池中代币总量) x 每小时借款费率 x 仓位规模
```

利用率越高,费率越高(双斜率模型)。这意味着当某资产被大量借出(如牛市中 SOL 多头拥挤)时,借款费可能飙升。

### 3.7 清算机制

**清算触发条件:**
```
保证金(扣除费用后) + 未实现盈利 - 未实现亏损 < 仓位规模 x 0.2%
```

即当有效保证金低于仓位规模的 0.2%(维持保证金率)时触发清算。

- 清算由 Keeper 网络自动执行
- 清算过程在单笔 Solana 交易中完成
- 清算费用归入 JLP 池
- 无保险基金机制 -- JLP 池本身承担所有对手方风险

### 4.1 完整费率结构

- Jupiter 研究团队每周发布风险报告,持续监控池健康状况

### 7.2 JLP 系统性风险

- **交易者集体盈利风险**: 在极端单边行情中(如 SOL 暴涨),多头交易者大幅盈利,JLP 作为对手方承担巨额亏损

- **流动性螺旋风险**: LP 亏损 -> LP 撤出 -> 流动性下降 -> 借款费暴涨 -> 交易者离开 -> 收入下降 -> 更多 LP 撤出
- **无保险基金**: 与 dYdX/Hyperliquid 不同,Jupiter 没有独立的保险基金,JLP 池直接承担所有风险

### 7.3 已知事件

- 截至目前,Jupiter Perps **没有发生重大安全事件或资金损失**
- 2025年初 Hyperliquid JELLY 事件后,Jupiter 团队公开表示其架构设计天然抵御此类攻击(因为 JLP 池作为对手方,不存在可被挤兑的做市商仓位)

### 7.4 核心局限性

| 局限 | 影响 |
|------|------|
| **交易对少(~50 对)** | 无法满足长尾资产交易需求,远少于 dYdX(235+)和 Hyperliquid(150+) |

## 2. 技术架构

### 2.1 Cosmos SDK 应用链

dYdX v4 是一条基于 Cosmos SDK 构建的 **Layer 1 主权区块链**,具有以下特征:

- **不依赖以太坊或任何外部 L1 执行或安全**
- 拥有自己的验证者网络和基础层逻辑
- 使用 CometBFT(前 Tendermint)共识
- 代码结构模块化: Markets 模块管理市场参数,Rewards 模块处理奖励分配等

### 2.2 验证者集

| 指标 | 数值 |
|------|------|
| 活跃验证者数量 | ~60 个 |
| 最大验证者数量 | 由 MaxValidators 参数决定 |
| 平均佣金率 | 6.08% |
| 最低佣金率 | 5% |
| 最高佣金率 | 100% |
| 验证者选择 | 按质押权重排名,前 N 名进入活跃集 |

验证者职责:
- 存储和维护内存中的链下订单簿
- 通过 Gossip 协议向其他验证者传播交易
- 通过共识过程产生新区块
- 执行订单匹配和结算

### 2.3 链下订单簿 + 链上结算(混合架构)

这是 dYdX v4 最核心的技术创新:

**链下部分(Off-Chain):**
- 订单簿完全存储在验证者的**内存中(In-Memory)**
- 订单的放置、撤销、修改不写入区块链
- 订单通过验证者间的 Gossip 网络传播
- 每秒可处理数千个订单操作

**链上部分(On-Chain):**
- 只有**匹配成功的交易(Fills)**才写入区块链
- 清算、资金费率结算、保险基金操作等链上执行
- 通过共识确保所有验证者对匹配结果达成一致

**优势**: 将高频订单操作保持在链下,仅将关键结算上链,大幅提升吞吐量。
**缺点**: 链下订单簿的透明性和可审计性不如 Hyperliquid 的完全链上方案。

### 2.4 CometBFT 共识

| 指标 | 数值 |
|------|------|
| 区块时间 | ~1 秒 |
| 交易最终性 | 1 区块(~1-2 秒含网络延迟) |

### 3.3 订单类型

| 订单类型 | 说明 |
|---------|------|
| **限价单(Limit Order)** | 传统限价单 |
| **市价单(Market Order)** | 以最优价格立即成交 |
| **止损单(Stop Market/Limit)** | 价格触发后以市价或限价执行 |
| **止盈单(Take Profit)** | 盈利目标价格平仓 |

### 3.4 MegaVault 系统

MegaVault 是 dYdX v4 的**协议级做市金库**,2024年末推出:

**工作原理:**
- 用户存入 USDC 到 MegaVault
- 资金自动分配到各市场作为做市流动性
- 收益来源: 做市 PnL、交易费收入分成
- 亏损也由存款人按比例承担

**关键数据:**
- TVL: 突破 **$8,000 万**(2025年)
- 上线 3 周内从 0 增长到 $4,000 万
- 初始 APR: 约 **50%**(上线初期)
- 当前 APR: 随 TVL 增长而下降
- 由 Gauntlet 提供风险管理和参数优化

**MegaVault 的战略意义:**
- 解决了 dYdX 作为订单簿 DEX 的**冷启动流动性问题**
- 使得新市场上线时自动有流动性
- 实现"即时市场上线"(Instant Market Listings)-- 任何人可以提议新市场,MegaVault 自动提供初始流动性

### 3.5 保证金模式

dYdX v4 同时支持**全仓(Cross Margin)和逐仓(Isolated Margin)**:

**全仓保证金(Cross Margin):**
- 默认模式,子账户 0
- 所有 Cross 类型的仓位共享保证金
- 一个仓位的盈利可以补贴另一个仓位的亏损
- 资金利用效率更高

**逐仓保证金(Isolated Margin):**
- 每个 Isolated 仓位占用一个独立子账户
- 保证金完全隔离,一个仓位清算不影响其他
- 子账户数量: 从 127 扩容到 **128,000 个**(v5.0.0 升级)
- 子账户 0-127 为父账户,128-128,000 为子账户

### 3.6 资金费率机制

dYdX 使用标准的**8 小时资金费率(Funding Rate)**机制:
- 正资金费率: 多头向空头支付
- 负资金费率: 空头向多头支付
- 目的: 使永续合约价格锚定现货价格
- 计算基于 Mark Price 与 Index Price 的偏差
- 与 CEX 机制一致,交易者可以利用资金费率进行套利

### 3.7 清算机制与保险基金

**清算流程:**
1. 当账户保证金率低于维持保证金要求时触发
2. 系统首先尝试降低仓位(部分清算)
3. 如果部分清算不足,则全部清算
4. 清算亏损首先由该市场的保险基金承担

**保险基金:**
- 每个 Isolated 市场有独立的保险基金
- Cross 市场共享保险基金
- 保险基金来源: 清算盈余、协议费用分配
- 与 Jupiter Perps 的 JLP 池直接承担风险不同,dYdX 有独立的风险缓冲层

## 4. 费率体系

### 4.1 Maker/Taker 费率分级

dYdX 使用标准的 Maker-Taker 费率模型,基于 30 天滚动交易量分级:

| 30天交易量 | Maker 费率 | Taker 费率 |
|-----------|-----------|-----------|
| < $1M | 0.020% | 0.050% |
| $1M - $5M | 0.015% | 0.040% |
| $5M - $10M | 0.010% | 0.035% |
| $10M - $50M | 0.005% | 0.030% |
| > $50M | **-0.011%(返佣)** | 0.020% |

**注意**: 具体费率可能随治理提案调整。以上为参考值。

### 4.2 特殊费率政策

- **BTC 和 SOL 永续合约: 零费用** -- 2025年通过社区投票取消了 BTC 和 SOL 市场的 Maker 和 Taker 费用
- **DYDX 质押折扣**: 质押 DYDX 代币可获得额外费率折扣(具体折扣通过治理调整)
- **交易奖励**: 交易者可获得 50% 的交易费返还(Trading Rewards 的 C 参数设为 0)

### 4.3 实际交易成本分析

考虑到交易奖励和质押折扣后,dYdX 的**有效交易成本**可以非常低:
- 高量 Maker: 实际获得返佣
- BTC/SOL 交易: 完全免费
- 一般 Taker: 0.05% 名义费率,扣除 50% 返还后约 0.025%

## 7. 风险与治理问题

### 7.1 市场份额下降原因分析

| 因素 | 详细分析 |
|------|---------|
| **v3 -> v4 迁移摩擦** | 用户需要桥接资产到新链,很多用户在此过程中流失到 Hyperliquid |
| **Hyperliquid 空投** | HYPE 代币空投将 31% 供应分配给早期用户,引爆社区热情 |
| **产品体验差距** | Hyperliquid 的 0.1s 延迟 + 零 Gas vs dYdX 的 1s 延迟 |
| **激励策略失误** | dYdX 依赖现金激励刺激交易量(wash trading 嫌疑),Hyperliquid 靠产品口碑增长 |
| **杠杆限制** | dYdX 最高 20x 杠杆,远低于竞品 |
| **流动性外迁** | 做市商和交易者跟随交易量迁移到 Hyperliquid |

### 7.2 治理争议

- **代币排放争议**: 社区围绕 DYDX 排放速度和分配比例有持续辩论
- **回购计划**: 从 25% 提升到 75% 协议费用用于回购,存在是否可持续的疑问
- **v3 关停**: 2024年10月关停 v3,2025年6月关闭桥接通道,部分 ethDYDX 持有者资产无法迁移
- **Treasury SubDAO**: 金库管理的透明度和效率受到社区质疑

### 7.3 安全事件

| 事件 | 时间 | 影响 |
|------|------|------|
| **$900 万交易漏洞** | 2023年11月 | 单一用户利用YFI市场低流动性进行价格操纵获利 $900 万 |
| **DNS 劫持攻击** | 2024年7月 | Squarespace OAuth 漏洞导致域名被劫持,2 用户损失 $31,000 |
| **供应链攻击(npm/PyPI)** | 2026年2月 | 官方 npm 和 PyPI 包被注入恶意代码,窃取钱包助记词并安装远程木马 |
| **早期供应链攻击** | 2022年9月 | 员工 npm 账户被劫持,恶意包窃取凭证 |

供应链攻击尤其令人担忧 -- 这意味着使用 dYdX SDK 的开发者和交易机器人运营者面临直接的资金安全风险。

### 7.4 验证者中心化风险

- 活跃验证者约 60 个,相比以太坊的数十万验证者,中心化程度较高
- 质押权重集中在少数大型验证者手中
- 但这是 Cosmos 生态链的通病,非 dYdX 独有

## 总体评价

**Hyperliquid** 是当之无愧的王者——70%市场份额、$8.44亿年收入、0.1s延迟。但JELLY事件暴露了"伪去中心化"本质(16验证者、81%基金会控制)，闭源代码也是长期隐患。

**dYdX v4** 是DeFi历史上最剧烈的领导者陨落之一——从73%到个位数。技术雄心(Cosmos应用链、235+交易对)未能转化为市场优势。核心问题：v3→v4迁移给了Hyperliquid窗口期 + 产品体验(延迟/杠杆/UX)落后 + 代币激励策略失败。
